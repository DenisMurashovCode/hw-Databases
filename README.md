# Домашнее задание к занятию «Базы данных, их типы» - Мурашов Денис

## Задание 1. СУБД

1.1. - Реляционная (PostgreSQL / MySQL)
Она гарантирует целостность данных через механизм транзакций (ACID), обеспечивает чёткую структуру таблиц с жесткой схемой. 

1.1.* - Можно использовать bcrypt API (например, argon2, libsodium). В PostgreSQL можно использовать расширение pgcrypto.

1.2. - Для лендингов: документо-ориентированная NoSQL СУБД (MongoDB). Лендинги имеют гибкую структуру данных, поэтому документо-ориентированная БД без жёсткой схемы подходит. MongoDB позволяет быстро вносить изменения, не затрагивая приложение, и горизонтально масштабируется под высокую нагрузку.
Для CRM: реляционная СУБД (PostgreSQL) или документо-ориентированная (MongoDB)
CRM-система требует хранения связанных данных о клиентах, сделках, задачах. Если структура данных стабильна — лучше реляционная СУБД из-за надежных связей и целостности. Если CRM меняется и нужна гибкость — MongoDB.

1.3. - Иерархическая NoSQL или реляционная СУБД. Если это компания, то она иерархична, там есть четкая структура, поэтому подойдет иерархическая модель. Но для простоты и понятности можно использовать реляционную СУБД (PostgreSQL).

1.3.* - Да, можно использовать PostgreSQL. Можно создать отдельную базу данных на том же сервере.

1.4. - Графовая NoSQL (Neo4j), так как это логистика, маршрутизация. Объекты - это узлы, а маршруты - ребра. Графовые СУБД поддерживают алгоритмы поиска кратчайшего пути, что обеспечивает высокую скорость работы со связями.

1.4.* - Для них лучше использовать отдельную реляционную СУБД (PostgreSQL), связанную с графовой СУБД, так как закупки оперируют структурированными данными.

1.5.* - Наверное можно было бы использовать PostgreSQL, но она все равно хуже в логистических задачах по сравнению с графовыми БД. Поэтому если и использовать, то как компромиссное решение. 

## Задание 2. Транзакции

1. Начало транзакции - система инициирует транзакцию в СУБД.
2. Проверка данных - проверяется корректность номера телефона и суммы пополнения, наличие счета плательщика с достаточным балансом.
3. Списание средств - с банковского счета списывается указанная сумма.
4. Зачисление средств - на счет абонента у оператора связи зачисляется сумма пополнения.
5. Фиксация транзакции - если все операции выполнены успешно, транзакция фиксируется.
6. Уведомление - пользователю отправляется подтверждение об успешном пополнении, появляется запись в истории операций.

## Задание 3. SQL vs NoSQL

1. Целостность данных (ACID) - транзакции обеспечивают атомарность, согласованность, изоляцию и долговечность.
2. Стандартизированный язык запросов (SQL) - единый язык работает во всех реляционных СУБД, снижает порог входа и упрощает перенос кода.
3. Сложные запросы и аналитика - поддержка JOIN, подзапросов, оконных функций и агрегаций, это позволяет строить сложные аналитические отчеты без переноса логики в приложение.
4. Нормализация и отсутствие дублирования - реляционная модель позволяет хранить данные без избыточности, что экономит место и упрощает обновление связанных записей.
5. реляционные СУБД существуют уже давно, имеют отлаженные механизмы резервного копирования, репликации, мониторинга и сообщество.

## Задание 4. Кластеры

ключевым критерием является горизонтальная масштабируемость и возможность параллельного выполнения вычислений. Реляционные СУБД плохо масштабируются горизонтально, поэтому лучше использовать NoSQL.

модель распределённых вычислений: Column-oriented СУБД + MapReduce

Column-oriented СУБД (Например, ClickHouse) - эффективно работает с аналитическими запросами над большими объемами данных, хранит данные по столбцам, что ускоряет агрегации.
Модель MapReduce - позволяет распределить вычисления по всем 1000 машинам: Map параллельно обрабатывает данные на каждом узле, Reduce агрегирует результаты. Это обеспечивает линейное масштабирование вычислительной мощности.

Все это подходит, так как задача вычислительная и при 1000 машинах нужна распределенная обработка с разбиением задачи на параллельные подзадачи.
